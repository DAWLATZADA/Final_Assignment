üìå Real Case: Centrum Air

Centrum Air sold more tickets than available seats. As a result, two passengers were unable to board their flight. This is a real-world example of a race condition caused by improper handling of a critical section.

‚ùå Problem: Overbooking (No Synchronization)

User A

  • Reads available seats = 1
  • Seat appears available
  • Proceeds to book

User B

  • Reads available seats = 1
  • Seat appears available
  • Proceeds to book

Result: Both users book the same seat ‚Üí OVERBOOKING

This happens because multiple users enter the critical section at the same time, causing a read/write conflict.

‚úÖ Solution: Synchronization (Mutex Lock)

Step 1

User A acquires the lock

Reads and updates available seats

Step 2

User B waits until the lock is released

Sees no seats available

Result: Only one user books ‚Üí NO OVERBOOKING

A mutex lock ensures that only one booking request can access the seat count at a time, protecting the critical section.
